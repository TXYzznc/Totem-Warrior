## 1. 玩家体验目标

**一句话**：10-15 分钟单局内，玩家对地图有**探索感**（不知道下一个房间是什么）、**决战感**（缩圈把所有人压向一个中心）、**主题独特感**（AI 废墟 / 外星侵蚀 / 病毒变异看起来不一样）。

三条体验目标拆解：

| 体验目标 | 设计手段 | 失败对立面 |
|---|---|---|
| **探索感** | BSP 保证每局房间布局不重复；小地图雾战遮挡，玩家靠脚探；关键点位隐藏在支路而非主路 | 地图每局形状一模一样，玩家"背路"到达关键点 |
| **决战感** | 缩圈中心仅在地图中央 1/3 内随机偏移，圈形压迫路线不可预判；Boss 固定出现在圈内安全区 | 圈心固定在地图正中，玩家第一局就能"预背圈路"；缩圈无法形成三方纠缠 |
| **主题独特感** | 房间 tileset / 装饰物 / 颜色方案完全来自 `MapThemeConfig.TerrainPoolId`；同一 BSP 骨架换皮后视觉截然不同 | 三主题换色但地形语言一模一样，视觉无差异感 |

**玩家学什么**：地图不是背景板，是博弈资源——工作室在哪里、精英怪偏置在哪里、圈心偏移方向，都是可以通过**主动探索**获得的信息优势。

**玩家感受什么**：每局落地时的"第一眼主题确认"（配色 + 环境音先于任何 UI）；探索未知房间时的轻度期待感；缩圈收紧时的逼迫感。

**玩家炫耀什么**：知道在哪个主题地图里精英怪密度高、工作室分布规律——这种空间知识是老玩家对新玩家的隐性优势。

## 2. 核心机制

### 2.1 地图规格

| 参数 | 推荐值 | 说明 |
|---|---|---|
| 地图尺寸 | **150 × 150 m** | 适配 10-15min 单局；50m 半径足够分区探索，不产生大量空跑；比 200m 减少约 44% 面积，缩圈压迫更早生效 |
| 主题数量 | 1 个 / 局 | 单一主题保证视觉一致性；混合主题留作 DLC 扩展位，本期不做 |
| 单局平均房间数 | 16-24 间 | BSP 分区深度 4 层，每区至少 1 间可进入房间；对应 150m 密度略高于 200m 版本 |
| 房间尺寸范围 | 8×8 m ~ 35×35 m | 小房间走廊连接感强，大房间用于 Boss 战场 / 开阔混战 |

### 2.2 生成算法 — BSP（二叉空间分割）

**选型理由**：

| 算法 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|
| **BSP（推荐）** | 生成快（< 1s）、房间不重叠、连通性保证、参数直觉 | 地图感"方正"，旋转对称略弱 | 有限时间预算、结构稳定、关键点位保底 |
| WFC（Wave Function Collapse） | 视觉更有机、tile 拼接自然 | 生成慢（可能 > 5s）、极端情况可能无解需回滚、关键节点保底逻辑复杂 | 美术主导的精雕关卡、离线预生成 |

**推荐 BSP** 的关键原因：单人项目首期，需要稳定可控的生成管线；关键点位（工作室 / 商人 / Boss 房）的保底逻辑在 BSP 树中天然可以"指定某层节点"，无需额外回溯修正。

**BSP 流程（4 步）**：

```
1. 分割：从 150×150 根节点开始，递归随机分割（水平/垂直均可）
   → 分割比例：0.45 ~ 0.55（防止极端细长房间）
   → 分割深度：4 层 → 生成 12-20 个叶节点

2. 填房：在每个叶节点内随机放置 1 间房间（留边距 2-4m 作走廊空间）

3. 连廊：自叶节点向上回溯，每对兄弟节点之间生成 1 条走廊
   → 走廊宽度：2-3m（BR 风格，允许追击与躲避）

4. 装饰：按 MapThemeConfig.TerrainPoolId 替换所有 tile
   → 走廊、房间地面、墙壁、顶部遮挡各走独立的 tile 池
```

### 2.3 关键点位保底逻辑

每局地图必须保证以下节点**至少存在一个**，且满足位置约束：

| 节点类型 | 数量 | 位置约束 | 保底方式 |
|---|---|---|---|
| **纹身师工作室** | 2-3 个 | 地图各象限各 1（不全堆一侧），距出生点 > 30m | BSP 分割后在 4 个象限叶节点中各选 1 间指定工作室 |
| **商人摊位** | 1-2 个 | 靠近地图中部偏外（方便中期转圈时顺路） | 在 BSP 第 2 层分区各指定 1 间 |
| **Boss 房** | 1 个 | 落在初始安全圈 1/3 ~ 2/3 区域内、房间尺寸 ≥ 25×25m | 最后生成阶段从大尺寸房间中筛选，BSP 为 Boss 预留最大叶节点 |
| **精英怪 Spawner** | 4-8 个 | 优先分布在缩圈路径边缘（距初始圈边界 20-40m）| SpawnBias = "ZoneEdge"，MapGenModule 在缩圈路径计算后追加 |
| **宝箱节点** | 12-20 个 | 全图分散，支路密度 > 主路密度（引导探索） | 支路房间内强制放置 1-2 个宝箱节点 |
| **环境叙事点位** | 0-2 个 | 不在主路，靠近盲路末端 | 从装饰物池随机选取，不强制必须有 |

**v2.1 移除**：圈外不再生成稀有怪或稀有宝箱节点。圈外区域仅作为伤害区，不提供任何额外奖励诱因——避免玩家因"圈外稀有"而作出高风险低收益的迂回行为，保持决策清晰。

**工作室节点与 09-NPC 系统耦合**：工作室节点只标记位置与类型，NPC 实例化由 NPCModule 订阅 `MapGeneratedEvent` 后消费 `RoomInfo.NodeType == TattooStudio`。

### 2.4 缩圈中心点算法

**目标**：圈心不固定，但也不能在地图角落（否则大量区域永远不经过）。

**v2.1 约束收窄**：圈心**仅在地图中央 1/3 内随机偏移**，相比 v2.0 的"中央 30%~70%"范围进一步收紧，消除极端角落情形。

**推荐算法**：

```
地图总尺寸：150×150m
中央 1/3 区域：x ∈ [50, 100]，y ∈ [50, 100]（各方向各 50m）

1. 在 [50, 100]×[50, 100] 内均匀随机采样圈心 (cx, cy)
2. 叠加 ±10m 小幅随机扰动，防止连续局圈心完全相同
   → 约束扰动后仍落在 [40, 110]×[40, 110] 内，避免极端偏移
3. 圈心位置在 Run 开始时确定，发布 ZoneShrinkPhaseEvent 时携带 NewCenter
```

这保证：
- 圈心始终在地图中央 1/3 区域（约 33% 面积），不会压到边缘
- 每局路径有差异，但整体战场重心稳定
- 地图四角始终有探索价值（圈到来之前可资源刷新），但不因圈心偏角落导致半个地图无意义

### 2.5 缩圈节奏（三段式，适配 10-15min）

> 设计原则：早期给玩家落地起步的时间窗，中期持续施压引导汇聚，后期急-慢节奏制造"最后冲刺感"。

```
总局时长：10-15 分钟（以 12 分钟中位值为基准设计）

早期（0-3 min）：大圈 + 慢缩
  → 初始安全圈半径：65m（覆盖约 70% 地图面积）
  → 目标：玩家落地后有充足时间刷 1-2 个纹身工作室，不被圈逼迫
  → 圈外伤害：2 HP/s（低，犯错有缓冲）

中期（3-9 min）：匀速施压 + 圈心锁定中央 1/3
  → Phase 1（3:00 开始）：60s 收缩至半径 45m
  → Phase 2（6:00 开始）：60s 收缩至半径 28m
  → 圈外伤害：Phase 1 = 5 HP/s，Phase 2 = 8 HP/s
  → 目标：强迫玩家离开边缘、形成中部汇聚；圈心锁中央确保两个 Phase 圈心连线不出现极端路程

后期（9-15 min）：急速收缩 → 慢收死圈
  → Phase 3（9:00 开始）：90s 急速收缩至半径 15m（"快进"感，紧迫！）
  → Phase 4（11:30 开始）：90-180s 缓慢收缩至半径 4m（死圈，耗尽剩余玩家）
  → 圈外伤害：Phase 3 = 15 HP/s，Phase 4 = 25 HP/s
  → 目标：Phase 3 的急速制造逃跑窗口感；Phase 4 的缓慢让最后存活者在极小空间内决出胜负
```

**节奏设计意图**：
- 急（Phase 3）→ 慢（Phase 4）的节奏对比，比纯匀速更有戏剧性
- Phase 3 玩家必须快速做决策，Phase 4 给予充足的"最终决战"时间
- 全程伤害倍率参考：基础 HP = 100；Phase 1 圈外 20s 致死，Phase 4 圈外 4s 致死

### 2.6 Spawn 节点同步调整（20 智能 + 29 轻量 + 1 玩家）

配合 v2.1 grill #11 的 AI 架构调整：

| AI 类型 | 数量 | Spawn 策略 | 地图绑定 |
|---|---|---|---|
| **SmartBot（智能 AI）** | 20 | 分散到 4 象限，每象限 5 个，出生点距玩家 > 30m | 订阅 `MapGeneratedEvent`，获取全图 RoomInfo[] 做长期规划 |
| **LightBot（轻量 AI）** | 29 | 靠近房间内 Spawner 节点分布，不做全局规划 | 仅订阅出生房间 + 相邻房间 NavMesh；视野外降频至 0.5Hz |
| **玩家** | 1 | 固定出生在地图中央偏外一侧（距边缘 20-25m）| — |

**150m 地图对 SmartBot 路径规划的影响**：
- 跨象限最大直线距离约 212m → 150m 地图约 106m，SmartBot 规划范围减小
- 早期（0-3min）SmartBot 有更大概率与玩家在工作室附近相遇，提升早期互动密度

### 2.7 生成时序

```
Run 开始
  │
  ├─ [1] MapGenModule.GenerateAsync() 启动（目标 ≤ 3s）
  │     ├─ BSP 分割（~0.08s，150m 比 200m 快约 40%）
  │     ├─ 房间 / 走廊生成（~0.15s）
  │     ├─ 关键节点保底分配（~0.1s）
  │     ├─ 圈心计算（~0.01s）
  │     ├─ Tile 替换（~0.5s，主题资源 I/O）
  │     └─ NavMesh Bake（~0.8-1.5s，150m 面积减少后显著缩短）
  │
  ├─ [2] MapGeneratedEvent 发布（携带 Seed / Theme / RoomInfo[]）
  │     ├─ SpawnerModule 订阅 → 实例化 EnemySpawnerNode（含 SmartBot/LightBot 分配）
  │     ├─ NPCModule 订阅 → 实例化工作室 / 商人 NPC
  │     ├─ EconomyModule 订阅 → 实例化宝箱节点
  │     └─ ZoneModule 订阅 → 初始化缩圈参数（三段式）
  │
  └─ [3] 玩家 / AI Bot 落地（UI 显示主题揭示字幕 1-2s）
```

## 3. 与其它系统的耦合

| 耦合系统 | 耦合方式 | 数据流 |
|---|---|---|
| **15-世界观与轻剧情** | 主题决定地形资源池，Map 不关心主题叙事内容 | `MapThemeConfig.TerrainPoolId` → MapGenModule（生成时读取） |
| **06-SpawnerModule** | MapGenModule 在房间信息中标记 `EnemySpawnerNode`（位置、Tier、SpawnBias）；SpawnerModule 消费 | `RoomInfo.SpawnerNodes[]` → SpawnerModule（订阅 MapGeneratedEvent 后） |
| **08-宝箱与探财节奏** | 地图中预埋宝箱节点位置；EconomyModule 实例化实体 | `RoomInfo.ChestNodes[]` → EconomyModule |
| **09-纹身师 / 商人 NPC** | 工作室 / 商人节点位置由地图生成确定；NPCModule 实例化 NPC 实体 | `RoomInfo.NodeType` → NPCModule |
| **11-怪物与 Boss** | Boss 房位置预留在 RoomInfo 中；精英怪 spawner 使用 `SpawnBias = "ZoneEdge"` | `RoomInfo.NodeType == BossRoom` → BossModule 监听 |
| **缩圈（ZoneModule）** | MapGenModule 计算圈心后写入，ZoneModule 订阅 MapGeneratedEvent 使用 | `MapGeneratedEvent.InitialZoneCenter` → ZoneModule |
| **小地图 / UI（13-UI-HUD）** | 地图生成后的 `RoomInfo[]` 供 MiniMapModule 构建雾战图层 | `RoomInfo.Bounds[]` → UIModule（MiniMapForm 初始化） |

**CONTRACT 事件对齐**（来自 CONTRACT.md §1.6）：

- `MapGeneratedEvent { int Seed; MapTheme Theme; List<RoomInfo> Rooms; }` — 地图生成完成，所有下游订阅此事件
- `RoomEnteredEvent { Actor Enterer; RoomRef Room; }` — 玩家 / AI 进入新房间，触发小地图解雾
- `ZoneShrinkPhaseEvent { int PhaseIndex; Vector2 NewCenter; float NewRadius; float Duration; }` — 缩圈阶段变化
- `OutsideZoneTickEvent { Actor Victim; float TickDamage; }` — 圈外伤害 tick

## 5. UX / UI 触点

### 5.1 小地图 — 雾战机制

- **初始状态**：小地图全黑，仅显示玩家出生点周围已探索区域。
- **解雾条件**：`RoomEnteredEvent` 触发时，对应房间的 `Bounds` 在小地图上显示轮廓（不显示房间内部细节，只显示"有这个房间"）。
- **关键节点图标**：以下节点在被解雾的房间内用图标标记（即使未进入也在进入相邻房间时部分提示）：
  - 纹身师工作室（纹身针图标）
  - 商人摊位（金币图标）
  - Boss 房（骷髅图标，Boss 刷新后变红）
  - 缩圈边界（橙色圆圈，实时更新）
- **精英怪位置**：不在小地图标记（精英是探索奖励，不提前暴露位置）。
- **图标颜色主题化**：小地图图标颜色跟随当前主题的 `HUDAccentColor`（来自 MapThemeConfig），不改功能语义。

### 5.2 主题标识

- Run 开始 1-2 秒内，屏幕中央显示主题字幕（由 15-世界观系统驱动，不由本系统负责）。
- 主题信息在 HUD 角落常驻 1 个小图标（主题 logo，色块 + 简短名称），方便截图分享与识别。

### 5.3 路径引导（无 HUD 箭头）

**不使用 minimap 箭头或屏幕边缘引导**。环境引导手段：

| 引导手段 | 实现方式 | 作用 |
|---|---|---|
| **Sightline 走廊** | 走廊端头必须能看到下一个房间入口的光源 | 玩家无需看 minimap 就能找到下一个房间 |
| **工作室光源偏暖** | 工作室入口放置暖色（橙/黄）灯光 cue，与一般走廊冷色灯区分 | 玩家远距离识别"那里有工作室"，无需 HUD 标记 |
| **Boss 房压迫感** | Boss 房入口有主题专属特效（电弧、菌丝蔓延、孢子雾），且是地图最大的空旷空间 | 玩家路过时能感知"这里有重要内容"，不需要 UI 说明 |
| **宝箱视觉差异** | 支路末端宝箱用高对比度发光轮廓（主题强调色），亮度高于环境光 | Breadcrumb 引导玩家往支路探索 |

## 6. AI 行为侧需求

### 6.1 智能 AI（SmartBot，20 个）— 需要全图布局

智能 AI 需要**长期空间规划**，必须在 Run 开始时获取完整地图信息：

- 订阅 `MapGeneratedEvent`，从 `RoomInfo[]` 中提取：
  - 工作室位置 → 纳入 `BotBuildPlanner.WorkshopPlan`（何时去刻纹身）
  - 商人位置 → 纳入消费决策
  - Boss 房位置 → 纳入晚期路径规划
  - 宝箱密集区 → 优先探索路径
- 缩圈中心点 (`MapGeneratedEvent.InitialZoneCenter`) → 用于路径规划"在被迫进圈之前到达目标点"的时间窗口。
- 150m 地图下，SmartBot 跨象限距离约 106m，路径规划计算量低于 v2.0，可适当提升规划频率（每 2s 一次，原来 3s）。

### 6.2 轻量 AI（LightBot，29 个）— 只需局部 NavMesh

轻量 AI 不做长期规划，只走局部 NavMesh：

- 不订阅 `MapGeneratedEvent` 全量数据。
- 只在当前房间 + 相邻房间内做寻路决策（DetectRange 为边界）。
- SpawnerModule 生成 LightBot 时直接给出初始房间引用，不需要全图数据。
- 视野外 LightBot 决策频率降至 0.5 Hz（参见 CONTRACT §四 LOD 规则）。

### 6.3 寻路性能约束

- 运行时 NavMesh Bake 目标 ≤ 2s（150m 地图预计 0.8-1.5s，比 200m 版本改善明显）。
- 静态房间 prefab 预烘焙 NavMesh（`NavMeshStatic = true`），运行时只合并连廊的动态部分。
- LightBot（Light tier）使用简化 grid 寻路（非完整 NavMesh），减少 Agent 数量对 NavMesh 的压力（参见 11-怪物 §七 §7.3）。

## 7. 风险与开放问题

### 7.1 BSP vs WFC — 已决策

**已选定 BSP**，理由见 §2.2。WFC 作为未来扩展可选项（可离线预生成地图库，运行时直接读取，规避生成时间问题），本期不实施。

### 7.2 生成耗时目标 — ≤ 3s

**v2.1 改善**：150m 地图面积比 200m 减少约 44%，NavMesh Bake 耗时预估从 1-2s 缩短至 0.8-1.5s，总生成时间目标 ≤ 2.5s（预留 0.5s 余量）。

**风险应对策略（优先级排序）**：

1. 所有 Room Prefab 预烘焙 NavMesh 静态部分（`NavMeshStatic = true`）；运行时只 bake 走廊拼接区域。
2. 如仍超时，采用分区懒加载：玩家未进入的区域延迟 bake，进入前 5s 触发。
3. 如仍超时，降低 NavMesh 精度（Agent 尺寸加倍，接受轻微卡墙），优先保证帧率。
4. 最终保底：离线预生成若干种子地图（50 个随机种子），运行时从库中采样，完全规避生成耗时。

### 7.3 缩圈急-慢节奏的感知校准

**风险**：Phase 3（急速 90s）对于行动力较差的玩家（纹身未刻满）可能产生挫败感，而非紧迫感。

**应对**：
- Phase 3 开始前 15s 给予明显的视觉/音效预警（缩圈闪烁 + 环境音急促化）
- Phase 3 期间圈边界发光加强（橙色 → 红色），强化视觉压迫感而非惩罚感
- playtest 验证：如果超过 40% 玩家在 Phase 3 圈外死亡，则将 Phase 3 持续时间延长至 120s

### 7.4 150m 地图的探索密度

**风险**：地图缩小后，20-29 个 AI + 1 玩家在 150m 地图早期（圈大时）有碰面概率上升，早期节奏可能过快，压缩刻纹身的安全窗口。

**应对**：
- 初始大圈（65m）覆盖约 70% 地图，AI 分散在 4 象限，玩家在一侧——早期碰面概率不高于 v2.0 的 200m 方案（相对密度相近）
- SmartBot 行为优先级：落地前 3min 以"前往工作室"为第一优先，而非"搜寻玩家"——降低早期 PvP 压力

### 7.5 主题地形的美术资源量

**风险**：3 套主题 × 完整 tile set（地面/墙/顶部/走廊/装饰）= 约 140-240 个 sprite/prefab（150m 减少约 20% 资源量级），对单人项目是显著工作量。

**应对**：
- 三套主题共享**相同的空间骨架**（BSP 参数一致，房间尺寸规格一致）。
- 只替换视觉层：color scheme + 关键装饰 prefab（3-5 个主题标志性物件即可建立差异感）。
- MVP 期先完成 `AI_RUINS` 一套完整 tileset；其余两套用不同色板的占位 asset，后期替换。

### 7.6 开放问题（需要后续闭环）

1. **房间布局是否需要"剧情房间"**（固定内容的特殊房间，如第一局教学房）？倾向：MVP 不做，后续可作为特殊 RoomTemplate 加入。
2. **是否需要垂直层差（高低差地形）**？当前设计假定平面地图，垂直差异仅靠视觉（高台 prefab）而非实际高度差。后续可加入但涉及 NavMesh 分层。
3. **地图种子是否对玩家展示**？展示种子→允许玩家分享/复现特定地图布局，对探索类玩家有社交价值。倾向：显示，在 Run 结束结算面板中展示。

> **设计哲学回应（v2.1）**：
>
> 150m 地图是"够用就好"的极简原则——单局 10-15min 内玩家探索的有效面积本就有限，更小的地图让缩圈压迫更早生效，让 AI 路径规划更轻量，让美术资源量更可控。圈心锁中央 1/3 是"保证决战在舞台中央"的导演意识：玩家不应因为圈心随机到角落而被迫在偏僻角落决出胜负。去掉圈外稀有的决策同样出于清晰性——圈内有价值、圈外有危险，这个认知模型对所有玩家都应是一致的，不需要"圈外高风险高收益"这层博弈噪音。
