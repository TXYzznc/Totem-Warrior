# 工作流迁移：从「工作/」5 Phase 到 openspec change 一站式

> 02-skill-routing-unification 后 agent 框架已经自动走 openspec 路径，「工作/」5 Phase 实际被绕过。本次完全退役「工作/」，把所有职责沉淀到 `openspec/changes/<NN-name>/` 单一目录树。

## 一、决策摘要

| 维度 | 决策 |
|---|---|
| 「工作/」整目录 | **删除**（含 3 个进行中老业务：STG游戏 / 背包示例 / 配置表，用户确认为「老业务，不用管」） |
| Phase 1 策划讨论 | 沉淀到 `openspec/changes/<name>/brainstorm.md`（非必须，但建议） |
| Phase 2 需求拆解 + 美术 | `proposal.md` + `art/requirements.md` + `art/prompts.md` |
| Phase 3 执行 | `design.md` + `tasks.md` + `CONTRACT.md`（多模块时）；`art/raw/` AI 出图源图 |
| Phase 4 测试 | `tests/plan.md` + `tests/results.md` + `tests/bugs.md`；测试代码进 `Assets/Tests/` |
| Phase 5 归档 | `openspec archive-change <name>` 自动迁移整个目录到 `openspec/changes/archive/`，**无需** `_v1/_v2` 版本号或「（已归档）」后缀 |
| ai-art SKILL 路径 | `工作/1.美术/...` + `工作/工作区/0.AI绘图...` → `openspec/changes/<name>/art/{requirements,prompts}.md` + `art/raw/` |
| 切图中间产物 `工作/工作区/1.切图完成/` | **退役**——切图直接进 `Assets/Resources/`，不留中间目录 |
| 美术大文件存储 | 直接 git（早期项目阶段够用；后期再考虑 LFS） |

## 二、目标 change 目录全景

```
openspec/changes/<NN-name>/
├─ .openspec.yaml
├─ proposal.md              ← Phase 2 需求拆解（openspec 原生）
├─ design.md                ← Phase 2/3 设计（openspec 原生）
├─ tasks.md                 ← Phase 3 执行追踪（openspec 原生）
├─ specs/<能力>/spec.md     ← openspec 原生
│
├─ brainstorm.md            ← Phase 1 策划讨论沉淀（新增约定）
├─ CONTRACT.md              ← 多模块全局契约（AGENTS.md 模式 5 依赖）
├─ art/                     ← Phase 2/3 美术（ai-art SKILL 落盘点）
│  ├─ requirements.md
│  ├─ prompts.md
│  └─ raw/
│     ├─ <资源名>.png
│     └─ 生成记录.md
└─ tests/                   ← Phase 4 测试（qa-engineer 落盘点）
   ├─ plan.md
   ├─ results.md
   └─ bugs.md
```

## 三、被否定的备选方案

### 备选 A：保留「工作/」+ 改 agent prompt 让它写到「工作/」里
- **否定理由**：违背 agent 框架的自动化心智；agent 不会主动写人工流程目录；且 openspec 已经自带 proposal/design/tasks artifact，强制写「工作/」是双写

### 备选 B：极简化「工作/」只留 1.策划 + 1.美术 + 工作区
- **否定理由**：仍是两套路径，归档时美术资源不能跟 openspec 同步走；用户已明确选择「全删」

### 备选 C：美术 PNG 走 git LFS
- **否定理由**：单人原型阶段过度配置；早期直接 git 即可；后期真膨胀再迁 LFS（`git filter-branch` 可追溯）

### 备选 D：保留 `_v1/_v2` 版本号机制
- **否定理由**：与 openspec「change 中心」心智模型冲突；迭代开新 change 更直观

### 备选 E：切图中间产物保留 `art/cut/`
- **否定理由**：切图是一次性 source → Asset 步骤，中间产物无价值

## 六、归档机制对比

| 旧机制（「工作/」） | 新机制（openspec） |
|---|---|
| `工作/4.已归档任务/NN.功能名_v1/` 手工 mkdir | `openspec archive-change <name>` 自动 mv 到 `openspec/changes/archive/<date>-<name>/` |
| 5 个阶段目录手工追加「（已归档）」后缀 | 整个 change 目录搬走，子目录 `art/` `tests/` `brainstorm.md` `CONTRACT.md` 自动随行 |
| `_v1 / _v2 / _v3` 版本号 | 迭代 = 新 change（如 `04-backpack-v2`） |

### codex-image-gen 与 ai-art 的协作

```
ai-art SKILL （前置）           codex-image-gen SKILL（后置）
─────────────────────         ────────────────────────────
- 沉淀 art/requirements.md     - openspec status 找 active change
- 沉淀 art/prompts.md          - 读 art/{requirements,prompts}.md
- 提示词模板路由              - codex exec 实际调用 imagegen
- 风格初始化                  - 落盘 art/raw/*.png
                              - 写 art/raw/生成记录.md
                              - 更新需求文件头部状态字段
```

两个 SKILL 都在 art-director / art-2d / art-ui 白名单；分别由 settings.json 的「初始化美术」hook 与「出图」hook 自动唤起。
