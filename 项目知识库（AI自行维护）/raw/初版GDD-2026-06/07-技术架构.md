## 二、纹身系统：数据驱动架构

### 核心问题
6 部位 × 7 颜色 × 8 图案 = **336 种组合**，如果每种都单独写代码 → 不可能维护。

### 解决方案
**用 21 个原子（ScriptableObject）+ 一套通用引擎** 在运行时合成效果。

### 架构概览

```
[数据层：ScriptableObject 配置]
   ├─ 6 部位资源（BodyPart）
   ├─ 7 颜色资源（Color）
   └─ 8 图案资源（Pattern）

[运行时层：核心代码]
   ├─ 事件总线（EventBus）
   ├─ 效果模块（EffectShape）
   ├─ 元素行为（ElementBehavior）
   └─ 纹身合成器（TattooComposer）

[联动层：后处理]
   └─ 联动管线（SynergyPipeline）
```

## 三、三大核心模块

### 模块 1：游戏事件总线

整个游戏只有一套有限事件（约 10-15 个）：

```
OnAttack       - 普攻命中时
OnCrit         - 暴击触发时
OnDamaged      - 自身受伤时
OnDodge        - 闪避成功时
OnKill         - 击杀时
OnSkillCast    - 释放技能时
OnMove         - 持续移动时
OnTick         - 每秒触发
OnHealthThreshold - 生命阈值触发
OnHeal         - 治疗时
（...扩展时再加）
```

**关键设计纪律**：
- 触发事件**不要膨胀**
- 新需求先看能否用现有事件 + 参数实现
- 如确需新事件，必须用于多个系统（避免私有事件）

### 模块 2：效果原子（EffectShape）

每种图案对应一个效果形状类（约 6-8 个）：

```
EffectShape (基类)
├─ SingleHit       - 单体一次性
├─ AOEBurst        - 范围爆发
├─ StackingMark    - 叠层引爆
├─ MultiHit        - 多段拆分
├─ ChainJump       - 连锁跳跃
├─ ProbBurst       - 概率触发
├─ TrailZone       - 留下轨迹
└─ SummonForm      - 召唤幻兽
```

每个类**只负责一种行为模板**，约 30-80 行代码。

### 模块 3：纹身合成器

运行时按合成规则将原子拼装成具体效果：

```
合成流程伪代码：

当 玩家装备纹身(部位, 颜色, 图案) 时:
    订阅 [部位.TriggerEvent] → 触发 ApplyEffect()

当 触发事件发生时:
    1. 取出图案的 EffectShape
    2. 取出颜色的 Element
    3. 取出部位的 TargetStat
    4. 计算基础数值 = Stat × ColorScaling × PatternMultiplier
    5. 应用联动倍率
    6. EffectShape.Apply(target, element, magnitude)
```

**这就是整个系统的核心运行逻辑。**

## 四、联动机制实现

联动是**纯函数式后处理**，不创建新效果：

```
联动管线伪代码：

每次纹身槽位变化时:
    遍历 6 个装备的纹身:
        计算其总倍率 = 1.0
        × ColorResonance(同色加成)
        × PatternEcho(同图案加成)
        × Symmetry(对称加成)
        × ColorCombo(颜色反应)
```

每个联动 = 一个小函数（20-40 行）。
新增联动 = 加一个函数，不动现有任何代码。

## 五、代码量估算

| 模块 | 估计代码量 |
|------|-----------|
| 事件总线 | 50-100 行 |
| 6-8 个 EffectShape 类 | 30-80 行 / 个，共 200-500 行 |
| 7 个 ElementBehavior（如有特殊行为） | 20-50 行 / 个，共 100-300 行 |
| 纹身合成器（装备、触发、计算） | 100-150 行 |
| 联动管线（4-5 个函数） | 20-40 行 / 个，共 100-200 行 |
| ScriptableObject 定义类 | 50-100 行（仅类骨架，资源由 Inspector 配置） |
| **总计** | **约 800-1200 行 C# 业务代码** |

**这是 Solo 开发完全可控的工程量。**

## 六、其他系统的实现要点

### 战斗系统
- 普攻 / 蓄力 / 闪避 / 技能 → 都通过事件总线对接纹身
- 武器决定普攻形态，但伤害判定与纹身的"附加效果"解耦
- 闪避帧使用 ScriptableObject 配置，便于调参

### 地图生成
- 全随机大地图 → 推荐采用**模块化拼接 + Wave Function Collapse / BSP** 算法
- 主题（AI / 外星 / 病毒）应作为生成参数，决定可用 prefab 集合
- 关键点位（纹身师工作室、BOSS 据点、商人）应**保底刷新**
- 性能预算：移动端需 mesh 合批、LOD、按距离剔除

### 缩圈机制
- 服务器权威决定圈心 / 圈速
- 客户端预测显示，避免视觉跳动
- 圈外伤害计算应客户端预判 + 服务器验证

### AI 玩家（人机）
- 行为树 / GOAP 实现
- 关键能力：探索、拾取、躲圈、PVE、PVP 决策
- 难度分级：根据当前对局人数和阶段动态调整 AI 强度

### 联机同步
- 玩家位置 / 状态：高频同步（~20Hz）
- 纹身配置 / 装备：低频同步（事件驱动）
- 敌人 AI：服务器权威 + 客户端预测渲染
- 战利品 / 宝箱：服务器权威

## 八、工程纪律

为保证 Solo 开发可控，遵守以下原则：

### 三条数据驱动纪律

1. **触发事件不要膨胀**
   - 控制在 10-15 个事件以内
   - 新需求优先复用现有事件

2. **效果形状不要膨胀**
   - 控制在 6-8 个 EffectShape 以内
   - 加新形状前先问"现有的能不能参数化覆盖"

3. **联动函数保持纯粹**
   - 只做"读状态 → 返回倍率"
   - 不直接修改纹身效果
   - 调试 / 平衡 / 回滚都简单

### 资源命名规范

```
Assets/
├─ Tattoos/
│  ├─ Parts/
│  │  ├─ Head.asset
│  │  ├─ Torso.asset
│  │  └─ ...
│  ├─ Colors/
│  │  ├─ Red.asset
│  │  └─ ...
│  └─ Patterns/
│     ├─ Line.asset
│     └─ ...
├─ Weapons/
├─ Skills/
├─ Maps/
│  ├─ Templates/
│  └─ Themes/
```

## 九、性能预算（移动端兼顾）

| 指标 | 目标 |
|------|------|
| 帧率 | PC 60+ FPS，移动 30+ FPS |
| 同屏物体数 | ≤ 200（含敌人、玩家、特效） |
| Draw Call | ≤ 100（合批后） |
| 纹理预算 | 单局 ≤ 200MB |
| 网络带宽 | 玩家上行 ≤ 50KB/s |

> 性能预算应在 MVP 阶段建立测试机制，避免后期重做。
