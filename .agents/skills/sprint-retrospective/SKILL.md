# 冲刺回顾会议


## 何时使用此技能

- **冲刺结束**：每个冲刺的最后阶段
- **项目里程碑**：主要版本发布后
- **团队出现问题**：需要立即开展回顾的时候

## 操作指南

### 步骤1：Start-Stop-Continue 模型

```markdown
## 回顾会议模板：Start-Stop-Continue

### START（开始做的事）
- 将每日站会缩短至5分钟以内
- 使用代码审查检查表
- 引入结对编程

### STOP（停止做的事）
- 周五下午部署（存在回滚风险）
- 频繁召开紧急会议
- 未做文档记录就添加功能

### CONTINUE（继续做的事）
- 每周技术分享会
- 自动化测试
- 透明化沟通

### 行动项
1. [ ] 将站会时间从9:00调整为9:30（团队负责人）
2. [ ] 编写代码审查检查表文档（开发人员A）
3. [ ] 发布禁止周五部署的规则通知（团队负责人）
```

### 步骤2：Mad-Sad-Glad 模型

```markdown
## 回顾会议：Mad-Sad-Glad

### MAD（让人愤怒的事）
- 部署后出现2次紧急Bug
- 需求频繁变更
- 测试环境不稳定

### SAD（让人遗憾的事）
- 代码审查时间不足
- 文档工作滞后
- 技术债务累积

### GLAD（让人满意的事）
- 新团队成员快速融入
- CI/CD流水线稳定运行
- 客户反馈积极正面

### 行动项
- 强化部署检查表
- 优化需求变更流程
- 每周周五预留文档编写时间
```

### 步骤3：4Ls 模型（Liked-Learned-Lacked-Longed For）

```markdown
## 回顾会议：4Ls

### LIKED（让人满意的事）
- 团队协作氛围良好
- 新技术栈引入成功

### LEARNED（学到的事）
- 用Docker Compose统一本地开发环境
- 用React Query优化服务器状态管理

### LACKED（欠缺的事）
- 性能测试环节
- 移动端适配

### LONGED FOR（渴望的事）
- 更优质的开发工具
- 外部培训机会

### 行动项
- 引入Lighthouse CI实现性能自动检测
- 编写响应式设计指南
```

## 输出格式

### 回顾会议文档

```markdown
# 第[N]次冲刺回顾会议
**日期**：2025-01-15
**参会人员**：团队成员A、B、C、D
**采用模型**：Start-Stop-Continue

## 做得好的方面
- 所有用户故事完成（交付速度：25点）
- 零Bug产生
- 团队氛围融洽

## 待改进的方面
- 技术调研耗时超出预期
- 因设计变更导致重复工作

## 行动项
1. [ ] 技术调研单独分配冲刺周期（团队负责人，截止到01/20）
2. [ ] 引入设计前置审查流程（设计师，截止到01/18）
3. [ ] 每周分享交付速度图表（Scrum Master）

## 核心指标
- 交付速度：25点
- 发现Bug数量：0
- 冲刺目标达成率：100%
```

## 约束规则

### 必须遵守的规则（MUST）

1. **安全空间**：营造无指责的沟通环境
2. **行动项**：必须具体且具备可执行性
3. **跟进机制**：在下一次回顾会议中检查行动项进展

### 禁止事项（MUST NOT）

1. **人身攻击**：聚焦流程改进而非针对个人
2. **过多行动项**：限制在2-3个以内

## 最佳实践

1. **时间限制**：会议时长控制在1小时以内
2. **轮换引导者**：由团队成员轮流主持会议
3. **庆祝成果**：共同庆祝团队取得的成功
